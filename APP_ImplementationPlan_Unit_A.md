# APP_ImplementationPlan_Unit_A: 순수 로직 고립 (상세 실행 계획)

본 계획은 `App.jsx` 내부에 산재한 비즈니스 로직을 독립된 모듈로 분리하여 코드 부피를 줄이고 유지보수성을 극대화하는 상세 로드맵입니다. 각 단계 완료 후 명시된 검증 계획을 수행하고, 수정 기록 지침에 따라 상세 변경 사항을 기록해야 합니다.

## 📌 핵심 원칙
1. **디자인 유지**: UI 디자인 및 CSS 클래스는 절대 수정하지 않습니다.
2. **MLX 전담**: 모든 코드 구조 재설계 및 로직 이전은 로컬 MLX 모델을 사용합니다.
3. **독립성 보장**: UI 구조나 상태 관리 체계를 변경하지 않고 오직 로직의 '위치'만 이동시킵니다.
4. **D2C 프롬프팅 적용**: MLX 추론 시 '생각 과정'을 생략하고 '코드 결과물'에 집중하는 구조화된 프롬프트를 사용하여 토큰 유실을 방지합니다.
5. **MLX 중심의 3단계 협업 고도화**: 
   - **1단계 (로컬 MLX)**: 컴포넌트 설계 및 Props 인터페이스 사전 정의, 안티패턴(Stale Closure, 리스너 중복 등) 정밀 분석, 국지적 코드 생성.
   - **2단계 (Cloud Gemini)**: MLX의 설계안을 기반으로 전역 시스템 의존성 검증, 최종 코드 조립 및 `App.jsx` 주입 연동.
   - **3단계 (사용자)**: MLX가 제안한 Edge Case 시나리오를 바탕으로 브라우저 UI 최종 QA 및 무결성 확정.

## 🛠 1단계: 내비게이션 가드 엔진 구축 (useNavigationGuard)
- **작업 내용**: 
  - `handlePopState`, `initDashboardHistory` 등 히스토리 관리 및 가드 로직을 상태 의존성(`Custom Hook`) 형태로 `src/hooks/useNavigationGuard.js`에 분리.
  - 앱 전체의 라우팅 안전망을 가장 먼저 확립하여 이후 비동기 작업 시 발생할 수 있는 라우팅 꼬임 현상 사전 차단.
- **검증 계획**:
  - [ ] **[AI 코드 분석] Stale Closure**: `viewState`가 최신 상태로 가드 내부에서 참조되도록 `Ref`를 활용한 동기화 구조가 구현되었는지 확인.
  - [ ] **[AI 코드 분석] Strict Mode 안전성 검증**: React Strict Mode 환경에서 `useEffect` 클린업 누락으로 인한 이벤트 리스너 중복 등록 및 메모리 누수 여부 확인.
  - [ ] **[브라우저 동작 확인] 뒤로가기 트랩 & 비동기 취소**: 비동기 작업 중 뒤로가기 시 안전하게 차단되며, 탭 이동 후 뒤로가기를 눌렀을 때 히스토리 스택이 의도대로 동작하는지 확인.

## 💡 로컬 모델(MLX-Qwen) 활용 가이드라인 및 레슨런
Unit B 작업 과정에서 도출된 로컬 추론형 모델(MLX-Qwen 27B 등)의 특징과 한계를 바탕으로, 본 프로젝트 전반에 적용할 프롬프팅 및 환경 설정 가이드라인입니다.

1. **프롬프팅 전략: 억압 대신 분리 유도 (Structured Output)**
   - **문제**: "생각하지 마(No thinking)", "JSON만 출력해"와 같은 강력한 부정어 제약(Negative Prompting)은 추론형 모델에서 쉽게 무시되며, 오히려 불필요한 텍스트 출력으로 이어집니다.
   - **해결**: 모델의 본능적인 '단계별 추론(Chain-of-Thought)'을 허용하되, `<think>생각 과정</think>`와 같이 별도 태그로 격리시키고 최종 결과물만 `<json>결과</json>` 등에 담도록 구조화된 출력을 요구해야 합니다.
2. **토큰 자원 할당: `max_tokens`의 여유로운 설정**
   - **문제**: 복잡한 코드 블록 분석 시, 모델이 추론을 작성하다가 `max_tokens`(예: 1200) 제한에 걸려 실제 결과물 출력이 중간에 잘리는 현상(Truncation)이 발생합니다.
   - **해결**: 코드 구조 분석이나 컴포넌트 Props 추출과 같은 고부하 태스크에서는 `max_tokens`를 최소 2000~4000 이상으로 넉넉하게 설정하여 모델이 작업을 끝마칠 수 있는 환경을 보장해야 합니다.
3. **태스크 세분화 (Single-Purpose Prompting)**
   - 한 번의 프롬프트로 "분석하고, 코드 짜고, JSON으로 포맷팅해"라고 요구하는 대신, "1. 상태만 추출해", "2. 함수만 추출해"처럼 로컬 모델의 처리 한계에 맞춰 지시를 잘게 쪼개는 것이 정확도를 높이는 지름길입니다.
